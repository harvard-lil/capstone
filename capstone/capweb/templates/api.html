{# ==============> OVERVIEW <============== #}

The Caselaw Access Project API, also known as CAPAPI, serves all official, published US court cases, and their related metadata, from 1658 to 2018. We digitized every case in this database during the first phase of the Caselaw Access Project, which we embarked upon in 2013. Before that, nearly every case was sitting in its original binding on Harvard Law Library shelves. Click here to learn more about the project.

CAPAPI is RESTful, and serves most of its data in JSON, but case text is also available in plain text, HTML, and XML. Though APIs are primarily useful to developers, anybody with a web browser can interact with it, though the interface is a bit clunky. (Perhaps, one day, some of you ambitious software developers in the audience will make a lovely, open-source case browsing application using this API. *nudge* *nudge*) Click here for some usage examples.

Our goal in undertaking this project was to make the data as open as possible. While there are restrictions that affect some users trying to access more than 500 cases per day, metadata for every case in every jurisdiction, and all cases from Illinois and Arkansas are available without restriction; you don't even need to be logged in or use an API key. Click here for more information on access restrictions, what type of users aren't subject to them, and how you can advocate to eliminate them in your legal jurisdiction.

{# ==============> RESTRICTIONS <============== #}

Restrictions

The agreement with our project partner, Ravel, stipulates that only bonafide academic researchers can access more than 500 cases per day, so all registered users are subject to this restriction. However, this stipulation does not apply to jurisdictions that make their cases freely available online in a citable, machine-readable format. We call these whitelisted jurisdictions. Currently, Illinois and Arkansas satisfy the requirements of the whitelist.

We encourage everybody to advocate for the expansion of that list! When a jurisdiction starts issuing cases in a machine readable, citable format, we can whitelist a jurisdiction in the API, and offer its cases to anonymous users in bulk. {# ==============> BEGINNERS <============== #}

Beginner's Introduction

Fundamentally, an API is no different from a regular website: A program on your computer sends a bit of data to a server, the server processes that data, and then sends a response. If you know how to read a URL, you can interact with web-based services in ways that aren't limited to clicking on the links and buttons on the screen.

Consider the following URL, which will perform a google search for the word "CAP." 

https://www.google.com/search?q=CAP

Let's break it down into its individual parts: 

https://

This first part tells your web browser which protocol to use: this isn't very important for our purposes, so we'll ignore it. 

www.google.com

The next part is a list of words, separated by periods, between the initial double-slash, and before the subsequent single slash. Many people refer to this as the domain, which is only partly true; the important takeaway here is that it points to a specific server, which is just another computer on the internet. 

/search

The next section, which is comprised of everything between the slash after the server name and the question mark, is called the path. It's called a path because, in the earlier days of the web, it was a path to a specific file on the web server. These days, it's more likely that the path will point to a specific endpoint.

You can think of an endpoint as a distinct part of a program, which could require specific inputs, and/or provide different results. For example, the "login" endpoint on a website might accept a valid username and a password for input, and return a message that you've successfully logged in. A "register" endpoint might accept various bits of identifying information, and return a screen that says your account was successfully registered.

Though there is only one part of this particular path, search, developers usually organize paths into hierarchical lists separated by slashes. Hypothetically, if the developers at Google decided that one generalized search endpoint wasn't sufficiently serving people who wanted to search for books or locations, they could implement more specific endpoints such as /search/books and /search/locations. 

?q=CAP

The final section of the URL is where you'll find the parameters, and is comprised of everything after the question mark. Parameters are a way of passing individual pieces of data, or values, to the endpoint to help it perform its job. In this case, the parameter tells the /search endpoint what to search for. Without this parameter, the response wouldn't be particularly useful.

A URL can contain many parameters, separated by ampersands, but in this instance, there is only one parameter: the rather cryptically named "q," which is short for "query," that has a value of "CAP." Parameter names are arbitrary— Google's developers could just as easily have set the parameter name to ?query=CAP, but decided that "q" would sufficiently communicate its purpose.

The Google developers designed their web search endpoint to accept other parameters, too. For example, there is an even more cryptically named parameter, 'tbs' which will limit the age of the documents returned in the search results. The parameters ?q=CAP&tbs=qdr:y will perform a web search for "CAP" and limit the results to documents less than a year old.

Glossary

This is a list of technical or project-specific terms we use in this docuemntation. These are stub definitions to help you get unstuck, but they should not be conisdered authoratative or complete. A quick Google search should provide more context for any of these terms.

Jurisdiction

Reporter
In this project, we use the term 'reporter' to refer to a reporter series. We'd consider F2d. a reporter.
Whitelisted
While most cases in the database are subject to a 500 case per day access limit, jurisdictions that publish their cases in a citable, machine-readable format are not subject to this limit. Click here for more information on access restrictions, what type of users aren't subject to them, and how you can advocate to eliminate them in your legal jurisdiction.
curl
curl is a simple command line tool for retrieving data over the internet. It's similar to a web browser in that it will retrieve the contents of a url, but it will dump the text contents to a terminal, rather than show a rendered version in a graphical browser window.
Command Line
This is the textual interface for interacting with a computer. Rather than interacting with the system through windows and mouse clicks, commands are typed and output is rendered in its textual form. On mac, the default Command Line program is Terminal. On Windows, the program is cmd.exe.
URL
A URL, or Uniform Resource Locator, is an internet address that generally contains a communication protocol, a server name, a path to a file or endpoint, and possibly parameters to pass to the endpoint.
API
API is an acronym for Application Programming Interface. Broadly, it is a way for one computer program to transfer data to another computer program. CAPAPI is a RESTful API designed with the intent of distributing court case data.
RESTful
A RESTful API is based on HTTP, and makes use of its built-in verbs(commands), such as GET and POST.
Server
A server is just a computer on the internet that was configured to respond to requests from other computers. A web server will respond to requests from a web browser. An email server will respond to requests from email programs, and or other email servers which are sending it messages.
Endpoint
You can think of an endpoint as a distinct part of a program, which could require specific inputs, and/or provide different results. For example, the "login" endpoint on a website might accept a valid username and a password for input, and return a message that you've successfully logged in. A "register" endpoint might accept various bits of identifying information, and return a screen that says your account was successfully registered.
URL Parameter
For our purposes, a parameter is just a piece of data with a label that can be passed to an endpoint in a web request.
